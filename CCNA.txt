Кадр Ethernet II, также известный как Ethernet DIX (Digital, Intel, Xerox), является наиболее широко используемым форматом Ethernet-кадра в современных сетях. Вот подробное описание структуры кадра Ethernet II, включая название, назначение и длину каждого поля:

1. **Преамбула (Preamble)**
   - **Длина:** 7 байтов (56 бит)
   - **Назначение:** Используется для синхронизации передающего и принимающего устройств. Содержит чередующуюся последовательность битов 1 и 0 (например, `10101010`), которая позволяет устройствам синхронизироваться перед передачей данных.

2. **Стартовый разделитель кадра (Start Frame Delimiter, SFD)**
   - **Длина:** 1 байт (8 бит)
   - **Назначение:** Указывает на конец преамбулы и начало данных. Значение этого поля — `10101011`, которое сигнализирует, что следующий байт — это начало кадра.

3. **MAC-адрес назначения (Destination MAC Address)**
   - **Длина:** 6 байтов (48 бит)
   - **Назначение:** Содержит MAC-адрес устройства, которому адресован кадр. Этот адрес должен быть уникальным для каждого устройства в сети.

4. **MAC-адрес источника (Source MAC Address)**
   - **Длина:** 6 байтов (48 бит)
   - **Назначение:** Содержит MAC-адрес устройства, которое отправляет кадр.

5. **Тип/Длина (Type/Length)**
   - **Длина:** 2 байта (16 бит)
   - **Назначение:** В Ethernet II это поле обычно используется для обозначения типа верхнего протокола (например, `0x0800` для IPv4, `0x0806` для ARP). Иногда его также называют "EtherType". В других вариантах Ethernet это поле может указывать длину поля данных.

6. **Поле данных (Payload/Data Field)**
   - **Длина:** От 46 до 1500 байтов
   - **Назначение:** Содержит передаваемые данные. Длина этого поля может варьироваться, но минимальный размер — 46 байтов, максимальный — 1500 байтов. Если длина данных меньше 46 байтов, то добавляется "набивка" (padding) для достижения минимального размера.
   Таким образом общий оазиер кадра (преамбулу не считают) равен 1518 байт (6+6+2+1500+4)

7. **Контрольная сумма кадра (Frame Check Sequence, FCS)**
   - **Длина:** 4 байта (32 бита)
   - **Назначение:** Используется для проверки целостности данных. Применяется алгоритм CRC-32 (циклический контрольный код), который позволяет обнаружить ошибки, возникшие во время передачи.

Это общая структура кадра Ethernet II. Различные расширения или варианты Ethernet могут иметь дополнительные поля или различия в структуре, но описанная выше структура является основной для классического Ethernet.

Пинг с компа 192.168.1.1 на комп 192.168.1.2 через новый свич, при пустых ARP таблицах на компах и пустой mac-table на свиче

- ПК1 планирует отослать пинг, зная то что адрес назначения в его сети (если не в своей сети, то он отправил бы его на мак дефолт гейтвея), когда пакет с 3г уровня спускается на второй, то комп проверяет соответствие Ip адреса и мак адреса в своей ARP таблице, но не находит соответствия, поэтому в действие вступает ARP протокол. Он буферизирует этот первый ICMP пакет, и отправляет броадкаст ARP запрос в сеть (кто с таким ip адресом, скажи свой mac). Этот ARP пакет уходит на свич. 
- Свич получает пакет, и добавляет мак адрес источника в свою mac таблицу (соответсвие мака и порта), и  рассылает полученный ARP кадр на все порты. 
- ПК2 получает кадр и добавляет данные в свою ARP таблицу и отправляет ARP ответ
- свич просто форвардит этот кадр на порт за которым находится ПК1, добавляя мак ПК2 в свою мак-таблицу
- кадр доходит до ПК1, он добавляет ip адрес ПК2 в свою ARP таблицу, ICMP пакет достается из буфера и уже имея соответсвие mac - ip в ARP таблице, формирует кадр и отправляет его на мак ПК2
- ПК2 получает кадр, декапсулирует его, видит что это пинг и отвечает на пинг

                Заголовок IPv4
    Version (Версия)
        Длина: 4 бита
        Указывает версию IP. Для IPv4 это значение равно 4.

    Internet Header Length (IHL) (Длина заголовка)
        Длина: 4 бита
        Указывает длину заголовка IP в 32-битных словах. Минимальное значение — 5 (для минимального заголовка 20 байт).

    Differentiated Services Field (Поле дифференцированных услуг) , Раньше называлось ToS (Type of Service)
        Длина: 8 бит
        Состоит из двух частей:
            Differentiated Services Code Point (DSCP) (6 бит): Определяет класс обслуживания и приоритет пакета.
            Explicit Congestion Notification (ECN) (2 бита): Уведомляет о перегрузке сети.

    Total Length (Общая длина)
        Длина: 16 бит
        Указывает общую длину IP-пакета, включая заголовок и данные. Максимальная длина — 65 535 байт.

    Identification (Идентификация)
        Длина: 16 бит
        Уникальный идентификатор пакета, используемый для фрагментации.

    Flags (Флаги)
        Длина: 3 бита
        Управляют фрагментацией. Содержат следующие флаги:
            Зарезервированный бит (должен быть 0)
            DF (Don't Fragment) — не фрагментировать
            MF (More Fragments) — есть еще фрагменты

    Fragment Offset (Смещение фрагмента)
        Длина: 13 бит
        Определяет положение фрагмента в оригинальном пакете, если пакет был фрагментирован.

    Time to Live (TTL) (Время жизни)
        Длина: 8 бит
        Определяет максимальное количество маршрутизаторов, через которые может пройти пакет. Значение уменьшается на 1 каждым маршрутизатором.

    Protocol (Протокол)
        Длина: 8 бит
        Указывает протокол верхнего уровня, которому передаются данные (например, TCP, UDP, ICMP).

    Header Checksum (Контрольная сумма заголовка)
        Длина: 16 бит
        Контрольная сумма для проверки целостности заголовка.

    Source Address (Адрес отправителя)
        Длина: 32 бита
        IP-адрес отправителя.

    Destination Address (Адрес получателя)
        Длина: 32 бита
        IP-адрес получателя.

    Options (Опции)
        Длина: Переменная
        Дополнительные параметры для управления маршрутизацией и безопасностью. Обычно не используется в стандартных пакетах.

    Padding (Заполнение)
        Длина: Переменная
        Дополняет заголовок до длины, кратной 32 битам.



                Классовая адресация

Вот основные классы адресации:

	•	Класс A: Диапазон от 0.0.0.0 до 127.255.255.255. Сеть определена первым октетом (8 бит), оставшиеся 24 бита используются для адресации внутри сети. Всего можно создать 128 сетей класса A, каждая из которых может содержать до 16 миллионов хостов.
	•	Класс B: Диапазон от 128.0.0.0 до 191.255.255.255. Первые 16 бит (два октета) определяют сеть, остальные 16 бит — адреса внутри сети. Доступно 16,384 сети класса B, каждая может содержать до 65,536 хостов.
	•	Класс C: Диапазон от 192.0.0.0 до 223.255.255.255. Первые 24 бита определяют сеть, последние 8 бит — адреса внутри сети. Это позволяет создавать 2,097,152 сети класса C, каждая из которых может содержать до 256 хостов.
	•	Класс D: Диапазон от 224.0.0.0 до 239.255.255.255. Этот класс предназначен для мультикастинга.
	•	Класс E: Диапазон от 240.0.0.0 до 255.255.255.255. Используется для экспериментальных целей и резервирования.

                Диапазоны частных IPv4-
                
Диапазоны частных IPv4-адресов определены стандартом RFC 1918. Эти диапазоны используются для адресов внутри локальных сетей и не маршрутизируются через Интернет. Вот три диапазона частных IPv4-адресов:

1. **10.0.0.0 – 10.255.255.255**
   - Маска подсети: 255.0.0.0 (или /8)

2. **172.16.0.0 – 172.31.255.255**
   - Маска подсети: 255.240.0.0 (или /12)

3. **192.168.0.0 – 192.168.255.255**
   - Маска подсети: 255.255.0.0 (или /16)

Эти диапазоны могут использоваться в любой локальной сети, и устройства внутри этой сети могут общаться друг с другом без необходимости использования публичных IP-адресов.

                DTP (dynamic tunking protocol)

Есть только на Цисках
Позволяет автоматически синхронизироыать смотрящие друг на друга порты свича CISCO, или access или trunk
show interfaces g0/0 switchport - показывет в каком режиме настроен порт
switchport nonegotiate - порт перестает отправляь DTP кадры
Перевод порта в access mode выключает DTP, но перевод порта в trunk mode - нет. 
По умолчанию транки на cisco имеют режим:
switchport trunk encapsulation negotiate 
поэтому прежде чем включить режим транк на порту, надо принудительно указать режим работы dot1q, иначе включится режим ISL, так как он в цисках имеет приоритетный выбор

DTP в целях безопасности надо откоючать и устанавливать все руками
switchport mode dynamic AUTO/DESIRABLE - это как раз и есть DTP. 
- desirable устанавливает транк с любым auto или desirable,а так же с прибитым гвоздями транком, 
- desirable устанавливает access только с прибитым гвоздями access портом
- auto - только если второй прибит гвоздями (транк с транком, аксесс с аксессом) - это passive способ
- два auto смотрящих друг на друга - образуют access

На старых свичах по умолчанию порты в режиме (administrative mode):
switchport mode dynamic desirable
На новых:
switchport mode dynamic auto

switchport nonegotiate - отключить DTP на конкретном порту


| Режим порта 1 | Режим порта 2 | Режим работы (access/trunk) |
|---------------|---------------|-----------------------------|
| access        | access        | access                      |
| access        | trunk         | incompatible                |
| access        | desirable     | access                      |
| access        | auto          | access                      |
| trunk         | trunk         | trunk                       |
| trunk         | desirable     | trunk                       |
| trunk         | auto          | trunk                       |
| desirable     | desirable     | trunk                       |
| desirable     | auto          | trunk                       |
| auto          | auto          | access                      |

**Примечания:**
- **access** — порт в этом режиме работает с одной VLAN и не передает теги VLAN.
- **trunk** — порт в этом режиме передает трафик для нескольких VLAN и добавляет теги VLAN.
- **desirable** — порт активно пытается установить соединение как trunk.
- **auto** — порт пассивно ожидает попытки установить соединение как trunk от другого порта.

Таким образом, если хотя бы один порт находится в режиме **desirable**, а другой в режиме **desirable** или **auto**, соединение будет установлено как trunk. Если оба порта находятся в режиме **auto**, соединение установится как access, так как ни один из портов не инициирует trunk-соединение.


                VTP (VLAN trunking protocol)

Свич сервер рассылает кадры VTP по транк портвм, синхронизируя вланы на всех свичах сети, которые в режиме сервер или клиент, но не в прозрачном режиме (только передача кадров VTP)
Они смотрят значение номера ревизии, и если он больше текущего, то обновляют свою таблицу вланов

show vtp status - показывает используемую версию, домнное имя, ревизию изменений вланов, кол-во вланов и режим работы свича (сервер, клиент, прозрачный)
По умолчанию свичи находятся в режиме server
Чтобы VTP работал обязательно надо указать доменное имя VTP
VTP версий 1 и 2 не поддерживают расширенный диапазон VLAN

vtp domain имя - установть доменное имя
Если свич вне домена получает VTP кадр, то он устанавливает себе имя домена и обновляет VLAN таблицу, если ревизия выше чем его

Опвсность!!: если к существующей сети подключить какой то старый свич с ревизией выше, то он затрет все VLANы у свичей и будет проблема

vtp mode server/client/transparent
В клиентсокм режиме нельзя создавать/удалять VLANы
ЧТобы сбросить ревизию надо изменить на неиспользуемый домен, или вклчить прозрачный режим

vtp version 2 - сменить режим VTP (VTP 2 поддерживает token ring, можно его не использовать)

                Spanning tree (STP) (протокол связующего дерева)

Как работает:
1) Сначала блокирует все порты и рассылает кадры BPDU
2) Определяет центральный свич (если свич получил BPDU с более меньшим bridge ID, то он сдает позицию и больше не является root bridgом). После схождения  (converge) топологии только root bridge рассылает BPDU, остальные только принимают
3) Определяет как с каждого свича лучше добраться до центрального и данный порт разблокируется, а если есть второй порт то он остается заблокированным

1) Как выбирается root коммутатор:
По наименьшему bridge ID (свич становится root bridge) - все порты на этом свиче разблокируются (состояние forwarding ports) (их роль - designated ports, это одна из ролей портов в STP)
Bridge ID состоит из 2х полей :
- bridge priority (16бит) - по умолч 32768 (можно менять на число 4096). Свич с самым маленьким Bridge Priority становится rootом
- mac address - если у всех свичей одинаковый bridge priority, то root bridge свич выбирается по наименьшему мак адресу (сравнивать цифры надо слева направо)

Порты к которым подсоединены конечные устройства тоже разблокируются и не участвуют в передачи кадров BPDU (Bridge PDU)

Если есть виланы, то root bridge свич выбирается для каждого влана и это могут быть разные свичи

 2) Далее каждый из не root свичей выбирает один порт, который называется root port и с которого наименьшая стоимоть пути (root path cost) до root bridgа
 Считать надо cost только исходящего интерфейса, не принимающего
 Каждый свич отсылает BPDU со значением cost своего интерфеса в сумме с costом всех исходящих интерфейсов через которые проходил кадр BPDU, отправленный root bridgом на противоположный интерфейс свича соседа. Таким образом сосед узнает стоимость пути через данный интерфейс 
 10 Mbps - 100
 100 Mbps - 19
 1Gbps - 4
 10Gbps - 2

Если свич соединен двумя линками с разными соседями с одинаковой стоимотью, то root портом становится порт соединенный с соcедом с наименьшим sender bridge ID (как считать bridge ID указано выше)
Если свич соединен с соседом двумя одинаковыми линками, то вибирается линк с соседом у которого порт с наименьшим sender port ID (priority number) (команда для просмотра этих номеров - show spanning-tree)

3) Разблокируется (designated) порт у свича с наименьшим cost до root bridga, если cost одинаковый, то на свиче с наименьшим bridge ID
Порт у оставшегося свича остается заблокированным (non-designated or blocking or alternate port)

Cостояния интерфейсов:

Blocking - не принимают и дропают сетевой трафик, но принимают BPDU. Не пересылают BPDU и NOT learn mac-addresses

Listening - в этом состоянии находятся root and designated ports.
15 cек, согласно forward delay
Не принимают трафик, просто получают и пересылают BPDU и NOT learn mac-addresses

Learning - состояние после Listening
15 сек 
Не принимают трафик, просто получают и пересылают BPDU
Но в этом состоянии порт LEARN mac-addresses!

Forwarding - работает нормально

RSTP (802.1w) Port State - Discarding -> Learning -> Forwarding

Роутер рассылает кадры LSA, чтобы все соседи постоили LSDB (базу данных)
После постороения LSDB роутеры используют алгоритм Дейкстры для построения наилучшего маршрута

Этапы:
1) Установление соседских отношений
2) Обмен LSA
3) Расчет наилучшего маршрута

Area - группа роутеров и линков, которые используют и делят между собой одну LSDB
Backbone area (0) - особая зона, с которой должны быть соединены все остальные areas
Внутренние роутеры - у которых все интерфесы внутри area
ABRs (area border routers) - роутеры между разными areas
BR (backbone router) - роутер соединенный с area 0, любой у которого хоть 1 интерфейс соединен с area 0
Intra-area route - маршрут внутри area
Interarea route - между areas

Установление соседства:
1) После вкл OSPF на интерфейсе  - down state 
2) Роутер отсылает Hello сообщение через равный hello timer (10s), это multicast 224.0.0.5 (всем роуетрам участвующим в OSPF). Сообщение инкапсулируется и ip заголовок содержит 89 в поле Тип Протокола. 
Hello сообщение содержит помимо прочего My RID: 1.1.1.1 и Neighbour RID: 0.0.0.0 (все нули так как мы пока не знаем о своем соседе)
Состояние OSPF в это время - init state
3) R2 добавляет инфу в свою таблицу соседей и отвечает так же Hello сообщением и R1 добавляет его в совю таблицу соседей
Потом R1 отсылает Hello сообщение уже с RID от R2 - это 2-way state
Тут идет DR/BDR election
4) Теперь роутеры готовы обменяться LSA пакетами и сформировать LSDB
Они выбирают кто будет master, а кто slave (exstart state)
Роутер с большим RID становится мастером и они обмениваются DBD (database description packets) в котором указано "Я буду мастер", но если RID R2 больше то он в ответ посылает свой DBD пакет "Нет, я"
5) Exchange state - обмениваются DBD содержащими базовую информацию об LSA (типо, я имею эти LSA, но основная инфа не передается)
Роутеры сравнивают данные об этих LSA, чтобы понять какие им надо
6) Loading state - роутер отсылает LSR (link state request) с инфой каких LSA у него нет (дай мне эти LSA), второй роутер отвечает ему пересылая эти LSA отправляя ему LSU (link state update) сообщение. После получения LSU роутер отвечает что получил эту инфу, отсылая сообщение LSAck. 
7)Теперь роутеры имеют одинаковую LSDB - full state
Они продолжают отсылать Hello пакеты каждые 10 сек для поддержания смежности
Как только роутер получает Hello сообщение он сбрасывает свой Dead timer (40s)

В OSPF (Open Shortest Path First) выбор DR (Designated Router) и BDR (Backup Designated Router) происходит на основе приоритета и Router ID (RID) следующим образом:

	1.	Приоритет: Каждый OSPF роутер имеет конфигурируемый параметр приоритета, который может принимать значение от 0 до 255. По умолчанию приоритет равен 1. Роутер с наивысшим приоритетом становится DR. Если значение приоритета установлено в 0, роутер не участвует в выборе DR/BDR.
        приоритет устанавливается непосредственно на интерфейсе командой

HSRP States

Initial - This is the state at the start. This state indicates that HSRP does not run. This state is entered through a configuration change or when an interface first becomes available.

Learn - The router has not determined the virtual IP address and has not yet seen an authenticated hello message from the active router. In this state, the router still waits to hear from the active router.

Listen - The router knows the virtual IP address, but the router is neither the active router nor the standby router. It listens for hello messages from those routers.

Speak - The router sends periodic hello messages and actively participates in the election of the active and/or standby router. A router cannot enter speak state unless the router has the virtual IP address.

Standby - The router is a candidate to become the next active router and sends periodic hello messages. With the exclusion of transient conditions, there is, at most, one router in the group in standby state.

Active - The router currently forwards packets that are sent to the group virtual MAC address. The router sends periodic hello messages. With the exclusion of transient conditions, there must be, at most, one router in active state in the group.

TCP - connection-oriented protocol (3-way handshake SYN, SYN+ACK, ACK), and 4-way FIN, ACK and FIN, ACK). Это флаги в TCP хэдере
TCP provide sequencing / acknowledgment - seq 10(рандомный номер) - seq50 + ack11 (номер ожидаемого сегмента) - seq11 + ack51 - и тд 
Таким образом сервер сообщает хосту (и наоборот) что получен каждый сегмент последовательности. Если подтверждение с ожидаемым номером сегмента не получено, то хост снова отправляет сегмент
TCP provides flow control - в заголовке есть поле - ОКНО, поэтому можно сразу по очереди прислать несколько сегментов и получить ack от сервера только на последний сегмент, так как ждать ack на каждый не эффективно

Таблица, которая наглядно отображает структуру заголовка TCP:

| Поле                | Длина (бит) | Описание                                               |
|---------------------|-------------|--------------------------------------------------------|
| Source Port         | 16          | Номер порта отправителя                                |
| Destination Port    | 16          | Номер порта получателя                                 |
| Sequence Number     | 32          | Порядковый номер байта данных                          |
| Acknowledgment Number | 32        | Номер подтверждения (только если флаг ACK установлен)  |
| Data Offset         | 4           | Смещение данных (указано в 32-битных словах)           |
| Reserved            | 3           | Зарезервировано для будущего использования (всегда 0)  |
| Flags               | 9           | Управляющие флаги (NS, CWR, ECE, URG, ACK, PSH, RST, SYN, FIN) |
| Window Size         | 16          | Размер окна (количество байт, которое может принять получатель) |
| Checksum            | 16          | Контрольная сумма (проверка целостности заголовка и данных) |
| Urgent Pointer      | 16          | Указатель на срочные данные (если установлен флаг URG)  |
| Options             | переменная  | Дополнительные параметры для расширения возможностей TCP |
| Padding             | переменная  | Дополнение до кратности 32 битам                       |

Управляющие флаги, каждый из которых занимает 1 бит:

    NS (Nonce Sum) (1 бит) - используется для защиты от повторных атак (RFC 3540).
    CWR (Congestion Window Reduced) (1 бит) - указывает на уменьшение размера окна из-за обнаружения перегрузки (RFC 3168).
    ECE (ECN-Echo) (1 бит) - используется для сигнализации о перегрузке сети (RFC 3168).
    URG (Urgent) (1 бит) - указывает на наличие данных с высоким приоритетом, определяемых полем Urgent Pointer.
    ACK (Acknowledgment) (1 бит) - указывает, что поле Acknowledgment Number содержит действительное значение.
    PSH (Push) (1 бит) - указывает, что данные должны быть немедленно переданы приложению.
    RST (Reset) (1 бит) - сбрасывает соединение.
    SYN (Synchronize) (1 бит) - инициирует соединение.
    FIN (Finish) (1 бит) - завершает соединение.

Формат заголовка UDP (User Datagram Protocol) намного проще, чем заголовок TCP. Он имеет фиксированную длину в 8 байт (64 бита) и включает следующие поля:

| Поле             | Длина (бит) | Описание                                        |
|------------------|-------------|-------------------------------------------------|
| Source Port      | 16          | Номер порта отправителя                         |
| Destination Port | 16          | Номер порта получателя                          |
| Length           | 16          | Длина всего датаграммы UDP (заголовок + данные) |
| Checksum         | 16          | Контрольная сумма (проверка целостности заголовка и данных) |

Эта таблица показывает, что заголовок UDP состоит всего из четырех полей, каждое из которых имеет фиксированную длину. UDP предоставляет простой способ передачи данных с минимальной нагрузкой на протокол, однако в нем отсутствуют механизмы управления потоком и подтверждения доставки, которые есть в TCP.

                        SNMP (simple network management protocol)

Протокол или даже фреймворк предназначенный для менеджмента (сбора инфы о дивайсе, управления и настройки) сетевого оборудования

- managed devices - те роутеры и свичи которыми мы управляем
- network managment station (NMS) - это SNMP сервер

- девайсы могут информировать сервер о каких либо измемнениях (напрмер если упал интерфейс)
- сервер может спрашивать девайсы о текущем статусе
- сервер может сказать девайсам поменять конфигурацию

Те или иные значения сохранены на девайсе в переменных, и сервер запрашивает или говорит изменить те или иные переменные (напрмер имя, ip адрес интерфейса и тд)

Виды SNMP сообщений:
- READ (get, genNext, getBulk)
- WRITE (set)
- NOTIFICATION (trap, inform) когда девайс посылает trap серверу, он не получают response в ответ (считаются ненадежным), в inform - получает (сервер отвечает что получчил сообщение)
- RESPONSE (response)

SNMP Agent слушает UDP 161
SNMP Manager - UDP 162

        Настройка агента на роутере (R1):

настраиваем community string (попросту говоря пароль для доступа) и уровень доступа (только чтение, или еще и возможность удаленной настройки):
snmp-server community ПАРОЛЬ1 ro - используя этот пароль SNMP сервер сможет только читать инфу с R1 (не сможет использовать SET сообщения)
snmp-server community ПАРОЛЬ2 rw - сможет уже настраивать девайс и читать

snmp-server host 192.168.1.1 version 2c ПАРОЛЬ1 - указваем сервер NMS, версию протокола и коммьюнити стринг (пароль) (кстати 2с - это номер версии - вторая си (community string) версия )

насраиваем какие конкретно traps будут отсылаться на червер:
snmp-server enable traps snmp linkdown linkup - настраиваем traps (если интерфейс упал или поднялся)
snmp-server enable traps config - если конфиг изменится, то traps будет отправлен на сервер

                                FTP and TFTP

        TFTP (Trivial FTP)
TFTP появился после FTP, он проще и позволяет только копировать файлы с сервера и на сервер
У него нет аутентификации, любой может скопировать файлы
Нет шифрования файлов, все передается в открытом виде
Сервер слушает UDP 69
Хотя TFTP использует UDP он сам предоставляет функцию надежной доставки сообщений. И сервер и клиент (в зависимости кто отправляет, а кто получает файл) отсылают Ack сообщение на каждый пакет полученных данных

Передача файлов по TFTP состоит из 3х фаз:
1) Connection
2) Data transfer
3) Connection termination
При первом подключении клиент использует 69 порт, но сервер отвечает с рандомного порта (TID - transfer identifier) и уже он используется далее

        FTP
Использует TCP 20 (ftp data connection - для передачи файлов) и 21 (ftp controll connection - используется для соединения и отсылки комманд)
Есть аутентификация, но тоже нет шифрования
Для безопасности можно использовать FTPS (FTP over SSL/TLS) - это апгрейд FTP
или SFTP (SSH FTP) - это новый протокол просто с похожим именем
FTP в отличие от TFTP может не только копировать файлы, но и перемещаться по директориям, удалять и тд

Сначала клиент инициализирует CONTROL CONNECTION и после установки соединения и отправки всех команд нужно допустим скачать файл. Существует 2 способа установки FTP data connection:
1) Active mode - клгда сервер инициирует TCP соединение, после чего FTP data connection считется установленным
2) Passive mode - когда клиент инициирует TCP сессию, это важно когда клиент находится за файерволом, блокирующим внешние подключения

        QoS (quality of service)
Три важных момента:
1) Bandwidth - полоса пропускания (н-р 100Mb/s)
2) Delay - задержка (one-way delay и two-way delay)
3) Jitter - разновидность one-way delay - разница между пакетами отправленными одним приложением (напрмер в основном пакеты доходят до адресата за 10 мс, а какой то идет 100 мс - это большая разница )
4) Loss - процунт пакет ов не дошедших до адресата
Для норм функционирования нужно:
- one-way delay 150мс или меньше
- jitter 30мс и менее
- loss 1% и менее

Хосты использующие TCP оспользуют sliding window чтобы повысить или уменьшить частоту отправки кадров. Хосты постоянно увеличивают частоту отправки кадров и как только происходит дропание какого либо пакета, то они уменьшают частоту, но потом увеличивают ее вновь и процесс дропания пакетов повторяется. Возникают волнообразные ухудшения связи.

Технологии качества обслуживания (Quality of Service, QoS) предназначены для управления сетевым трафиком с целью обеспечения стабильности, предсказуемости и контроля при передаче данных. Вот краткое описание основных методов QoS:

1. CBWFQ (Class-Based Weighted Fair Queuing)
CBWFQ (Классифицированное взвешенное справедливое очередование) расширяет возможности WFQ путем классификации трафика на основе различных параметров (например, протокола, адресов, портов и т.д.). Для каждой классифицированной очереди назначается вес, определяющий долю пропускной способности, которая будет выделена этой очереди. Это позволяет более гибко распределять сетевые ресурсы между различными типами трафика.

2. CQ (Custom Queuing)
CQ (Пользовательское очередование) позволяет администратору сети вручную настраивать количество и размер очередей для различных типов трафика. В этом методе каждому классу трафика выделяется фиксированное количество пакетов (байт), которые могут быть отправлены в одном цикле. Это полезно для обеспечения предсказуемого уровня обслуживания для определенных типов трафика, таких как голосовой или видеотрафик.

3. FIFO (First In, First Out)
FIFO (Первым пришел – первым вышел) — это самый простой метод очередования, где пакеты обрабатываются в порядке их поступления. Данный метод не различает типы трафика, не предоставляет приоритетов и не обеспечивает гарантированную пропускную способность для критичных приложений. Он подходит для простых сетей без сложных требований к качеству обслуживания.

4. PQ (Priority Queuing)
PQ (Приоритетное очередование) разделяет трафик на несколько очередей, каждая из которых имеет свой приоритет. Пакеты в очереди с высоким приоритетом обрабатываются первыми, что позволяет обеспечивать минимальные задержки для критичных данных. Однако этот метод может привести к голоданию очередей с низким приоритетом, если высокоприоритетный трафик доминирует в сети. Испоьзует 4 очереди (high, medium, low, veru low)

5. WFQ (Weighted Fair Queuing)
WFQ (Взвешенное справедливое очередование) обеспечивает более справедливое распределение сетевых ресурсов между разными потоками трафика. Каждой очереди назначается вес, который определяет долю пропускной способности, выделяемую этой очереди. Это позволяет гарантировать, что критичный трафик будет получать необходимую пропускную способность, при этом предотвращая голодание других потоков.


    CBWGQ : provides guaranteed bandwidth to a specified class of traffic
    CQ : services a specified number of bytes in one queue before continuing to the next queue
    FIFO : uses store-and-forward queuing
    PQ : places packets into one of four priority-based queues
    WFQ : provides minimum guaranteed bandwidth to one or more flows

В сетях, поддерживающих качество обслуживания (QoS), используются различные механизмы для управления трафиком. Вот основные из них:

        ### Traffic Shaping

**Traffic Shaping** (Формирование трафика) регулирует отправку данных с целью предотвращения перегрузок в сети. Оно "выравнивает" поток трафика до заранее определенной скорости, буферизуя пакеты, если скорость передачи превышает заданный лимит.

- **Принцип работы**: Пакеты, превышающие лимит, буферизуются и отправляются позже.
- **Использование**: Применяется для обеспечения соблюдения соглашений об уровне обслуживания (SLA) и для управления трафиком, который выходит за пределы сети.



        ### Traffic Policing

**Traffic Policing** (Полицирование трафика) контролирует скорость входящего и исходящего трафика, отбрасывая или маркируя пакеты, которые превышают заданные лимиты.

- **Принцип работы**: Пакеты, превышающие лимит, могут быть отбрасываны или помечены для более низкого приоритета.
- **Использование**: Применяется для ограничения скорости трафика и предотвращения перегрузок в сети.



        ### Class-Based Weighted Fair Queuing (CBWFQ)

**CBWFQ** предоставляет гарантированную пропускную способность для различных классов трафика, определенных в policy map.

- **Принцип работы**: Каждый класс трафика получает определенную полосу пропускания, остальные пакеты распределяются пропорционально их весу.
- **Использование**: Применяется для приоритизации различных типов трафика (например, голос, видео, данные).



        ### Low Latency Queuing (LLQ)

**LLQ** добавляет строгий приоритетный механизм в CBWFQ, позволяющий выделить полосу пропускания для трафика с низкой задержкой, такого как голос и видео.

- **Принцип работы**: Трафик с высоким приоритетом (например, голос) обслуживается первым, до обработки других очередей.
- **Использование**: Применяется для трафика, чувствительного к задержке, такого как VoIP.



### Другие политики в QoS

1. **Weighted Random Early Detection (WRED)**:
   - Используется для предотвращения перегрузок путем случайного отбрасывания пакетов до достижения полного заполнения очереди. Чем выше приоритет пакета, тем ниже вероятность его дропа. Таким образом он дропает пакеты с низким приоритетом, предотвращая заполнение очереди, чтобы потом не дропать пакеты с высоким приоритетом. 

WRED (Weighted Random Early Detection) — это механизм управления очередями (queuing) в сетях, который используется для предотвращения перегрузок и обеспечения качества обслуживания (QoS). WRED является усовершенствованной версией RED (Random Early Detection) и позволяет дифференцировать трафик по приоритетам.

        ### Основные принципы работы WRED:
        1. **Предотвращение перегрузок**: WRED предотвращает перегрузки сети путем раннего сброса пакетов. Вместо того чтобы сбрасывать пакеты только при переполнении буфера, WRED начинает сбрасывать пакеты заранее, когда использование буфера достигает определенного уровня. Это помогает уменьшить вероятность возникновения ситуаций, когда буфер полностью заполняется и все пакеты начинают сбрасываться.

        2. **Взвешенный подход**: В отличие от RED, WRED позволяет учитывать различные классы трафика с разными приоритетами. Взвешивание означает, что пакеты с низким приоритетом имеют больше шансов быть сброшенными в случае перегрузки, чем пакеты с высоким приоритетом.

        ### Как работает WRED:
        1. **Пороговые значения**: WRED настраивается с использованием нескольких пороговых значений, определяющих начало и конец процесса сброса пакетов. Эти пороговые значения включают минимальный порог (min-threshold) и максимальный порог (max-threshold). Когда использование буфера находится ниже минимального порога, пакеты не сбрасываются. Когда использование буфера превышает максимальный порог, все новые пакеты сбрасываются. В промежутке между этими порогами пакеты сбрасываются с определенной вероятностью.

        2. **Вероятностный сброс**: Если уровень заполнения буфера находится между минимальным и максимальным порогами, WRED начинает сбрасывать пакеты с некоторой вероятностью, которая увеличивается по мере приближения к максимальному порогу. Это позволяет постепенно уменьшать нагрузку на сеть, не вызывая резких скачков в сбросе пакетов.

        3. **Дифференциация трафика**: WRED может быть настроен на основе различных классов трафика с использованием меток DSCP (Differentiated Services Code Point) или IP Precedence. Например, трафик с высоким приоритетом (например, голосовой или видео трафик) может иметь более высокие пороговые значения, что снижает вероятность его сброса по сравнению с трафиком с низким приоритетом (например, не критичные данные).

        ### Преимущества WRED:
        - **Стабилизация очередей**: Предотвращение резкого переполнения очередей и улучшение общей производительности сети.
        - **Приоритет трафика**: Дифференциация и защита важного трафика от сбросов.
        - **Уменьшение задержек**: Раннее обнаружение и предотвращение перегрузок помогает снизить задержки и вариабельность задержек (джиттер).

        ### Пример настройки WRED на Cisco:
        ```cisco
        class-map match-any HIGH_PRIORITY
        match dscp ef
        match dscp af4
        !
        policy-map WRED_POLICY
        class HIGH_PRIORITY
        random-detect dscp-based
        random-detect dscp 46 20 40 10
        random-detect dscp 34 30 50 5
        !
        interface GigabitEthernet0/1
        service-policy output WRED_POLICY
        ```
        В этом примере создается класс трафика с высоким приоритетом, настраивается политика WRED для этого класса и применяется к интерфейсу.

        Таким образом, WRED является важным инструментом для управления перегрузками и обеспечения QoS в современных сетях, позволяя эффективно управлять очередями и приоритетами трафика.

2. **Class of Service (CoS)**:
   - Помечает трафик на канальном уровне (Layer 2), часто используется в сетях Ethernet и для MPLS.


3. **Differentiated Services Code Point (DSCP)**:
   - Маркировка трафика на сетевом уровне (Layer 3) для приоритизации трафика в IP-сетях.


Эти политики и механизмы позволяют сетевым администраторам эффективно управлять и оптимизировать трафик, обеспечивая соответствие требованиям различных типов приложений и сервисов.

Dos (DDos) - это TCP SYN flood - т.е. атакующий шлет TCP SYN запросы на установление 3х стороннего рукопожатия при установлдении TCP сессии и сервер отвечает пакетом SYN-ACK и вынужден ждать ответа от атакующего, который он не получит. Таким образом сервер достигает максимального количества TCP соединений и не может реагировать на легитимные запросы в подклбючении к сераеру.
Ответы от сервера не достигают атакующего, потому что он подделывает свой ip адрес.

Amplification - смысл как и у reflection attack, но атакующий шлет мало трафика на DNS, но с таким запросом, что DNS отвкечает большим количесвом трафика

Man-in-the-middle - например arp spoofing (arp poisoning) - жертва отсылает arp запрос - У кого такой IP адрес? Сервер отвечает жертве и сразу после этого атакующий отвечает жертве и запись в ARP таблице жерты перезаписывается.
Теперь когда жертва отсылает трафик на сервер, то его получает атакующий, читает (модифицирует) и пересылает на сервер.

Reconnaisance attack - разведка и сбор сведений об объекте атаки 

Malware - virus, worms, trojan horses

Social engeneering - fishing (рассылка например писем от типо доверенной тебе организации) (spear fishing - на сотрудников определенной комапании, whaling - на президента компании, vishing - телефонный фишинг, smishong - смс фишинг)

        Triple A

Организации обычно используют AAA серверы для организации этих моментов

- TACACS+ (это цисковский протокол, udp 49)

Аутентификация и авторизация являются двумя ключевыми аспектами управления доступом, и они различаются по своему назначению и реализации в протоколах RADIUS и TACACS+.

### Аутентификация (Authentication)
Аутентификация — это процесс проверки подлинности пользователя или устройства, т.е. подтверждение того, что субъект действительно тот, за кого себя выдаёт.

### Авторизация (Authorization)
Авторизация — это процесс определения прав и привилегий пользователя или устройства, т.е. разрешение на выполнение определённых действий или доступ к определённым ресурсам после успешной аутентификации.

### Различия в RADIUS и TACACS+
**RADIUS (Remote Authentication Dial-In User Service):**
1. **Аутентификация и авторизация**:
   - В RADIUS аутентификация и авторизация обычно объединены в одном шаге. Когда пользователь аутентифицируется, сервер RADIUS одновременно отправляет информацию о правах доступа.
   - RADIUS использует UDP и чаще применяется для сетевого доступа, таких как VPN, Wi-Fi, и коммутируемый доступ.

2. **Протоколы и шифрование**:
   - RADIUS использует UDP-порты 1812 (аутентификация) и 1813 (учет), хотя также могут использоваться порты 1645 и 1646 (исторически).
   - В RADIUS только пароль передается зашифрованным (используется MD5), в то время как остальные данные аутентификации и авторизации передаются в незашифрованном виде, что делает его менее безопасным по сравнению с TACACS+.

**TACACS+ (Terminal Access Controller Access-Control System Plus):**
1. **Аутентификация и авторизация**:
   - TACACS+ разделяет процессы аутентификации и авторизации. Они могут обрабатываться отдельно, что обеспечивает более гибкое управление доступом.
   - TACACS+ используется для доступа к устройствам управления сетью (например, маршрутизаторам и коммутаторам).

2. **Протоколы и шифрование**:
   - TACACS+ использует TCP-порт 49.
   - TACACS+ шифрует весь трафик между клиентом и сервером, включая пароли и другие данные, что обеспечивает более высокий уровень безопасности.

### Применение
- **RADIUS**:
  - Часто используется для удаленного доступа, таких как VPN, беспроводные сети и диал-ап-сервисы.
  - Хорошо подходит для ситуаций, где требуется одновременная аутентификация и авторизация.

- **TACACS+**:
  - Преимущественно используется для доступа к сетевым устройствам и управления ими.
  - Подходит для сценариев, где требуется более детальное и гибкое управление авторизацией и отдельная аутентификация.

                                        DHCP Snooping

Это фича СВИЧЕЙ, которая позволяет фильтровать DHCP сообщения полуенные с недоверенных портов

Сообщения DHCP севера:
- offer
- ack 
- nak - противоположность ack, чтобы отклонить запрос клиента

Сообщения DHCP клмента:
- discover
- request
- release
- decline - чтобы отклонить ip адрес предложенный сервером

Существует 2 вида портов, доверенные и недоверенные.
Недоверенные - это к которым присоединены конечные пользователи, или которые сморят вниз, доверенные - те которые смотрят наверх и к котрыи присоединен DHCP сервер.
Если любое DHCP сообщение получено с доверенного порта, то оно не проверяется и пропускается. Если с недоверенного, то:
- если это сообщение от DHCP сервера, то оно отклоняется, так как Сервер не должен быть подключен к порту смотрящему вниз и значит он зловредный
- если от клиента, то проверяется:
        - если это DISCOVER or REQUEST, то проверяется соответствие поля мак-адрес источника и поля CHADDR в самом DHCP сообщении. И там и там маки должны совпадать. Это сделано чтобы не допустить пересылку злоумышленником запроса от клиента (man-in-the-middle)
        - усли это RELEASE or DECLINE, то проверяется чтобы IP адрес источника этого сообщения и интерфейс соответствовали записи в таблице Snooping bindind table (show ip dhcp snooping binding).

                        Dynamic ARP inspection

DAI это фича свичей позволяющая фильтровать ARP сообзения, полученные с недоверенных портов

Gratuitous ARP - некоорые хосты отсылают ARP request, но им не требуется ответа, так как в ip адресе назначения указан его же адрес, а просто чтобы другие хосты узнали его мак и ip
Все порты по дефолту - недоверенные
Все порты соединенные с роутером или свичем надо настроить как доверенные, а с конечными хостами - оставить недоверенными
Так же как и при DHCP снупинге все порты по умолчанию считаются недоверенными, и надо настроить доверенные порты. НО можно просто аплинки сделать доверенными, а довнлинки - недоверенными, как и при настройке DHCP snooping.
DAI проверяет мак адрес отправителя ARP запроса на untrusted порте и его ip адрес согдасно таблице DHCP snooping binding table и если все соответствует, то запрос проходит через интерфейс. То есть DAI полагается на DHCP snooping, но есть и другой метод: 
- использование в ручную настроенных ARP ACLs в которых определенному маку соответствует определенный IP (это особенно нужно, если хост имеет статический ip и значит его IP нет в таблице DHCP snooping binding table)

Данные IP пакета шифруются, затем инкапсултируются в IPsec VPN заголовок и затем уже в обычный IP заголовок и отправляются в сеть 
- IPSec не поддерживает broadcast and multicast (но это решается с помощью GRE over IPsec)
  не зашифрованные данные инкапсулируются GRE заголовком, затем IP заголовком, далее шифруются IPsec и инкапсулируются в IPsec VPN заголовок и снова в IP заголовок
  When a site-to-site VPN is configured, IPsec tunnel mode with ESP (Encapsulating Security Payload) provides encapsulation and encryption of the entire original IP packet. In this mode, the entire IP packet, including the original IP header and payload, is encapsulated within a new IP packet with a new IP header added by the VPN gateway. The original packet is encrypted, ensuring confidentiality, and the new IP header allows the encrypted packet to be routed over the public internet securely to the other VPN gateway, where it is decrypted and forwarded to its final destination.
        Authentication Header (AH)
        Encapsulating Security Payload (ESP)


IPsec (Internet Protocol Security) is a suite of protocols used for securing internet protocol (IP) communications. The two protocols within the IPsec suite are:

    Authentication Header (AH): AH provides data authentication and integrity protection for IP packets, but does not provide encryption. AH calculates a hash value over the IP packet and some additional data, and places the result in a new header that is added to the packet.

    Encapsulating Security Payload (ESP): ESP provides both encryption and data authentication for IP packets. ESP encrypts the original IP packet and places it inside a new packet, along with a new ESP header that provides data authentication and integrity protection.

Both AH and ESP protocols can be used alone or in combination to provide secure communication between two network devices. AH and ESP are often used in conjunction with the Internet Key Exchange (IKE) protocol, which is used to negotiate and establish security associations (SAs) between devices.

                                        WI-FI (IEEE 802.11)

- все девайсы получают кадры при отправке точкой доступа, как будто она - хаб
- CSMA/CA - девайс ждем пока среда освободится и только потом отправляет кадр
- absorption (поглощение), reflection (отражение), refraction (преломление), diffraction (вокруг препятствий), scattering (рассеивание из за пыли, смога, неровных поверхностей)
- interference (если совпадают по фазе, то усиливаются)

Характеристики волн:

- frequency - расстояние между гребнями и впадинами

Wi-Fi 6 (802.11ax) так же использует 6 GHz band

В диапазоне 2.4 GHZ есть 3 непересекающихся кнала 1, 6, 11
В диапазоне 5 GHz используются непересекающиеся каналы

3 service sets:
- Independent
- Infrastructure
- Mesh 
Каждая service set имеет свой SSID (человекочитаемое имя сети, не обязательно уникальное)

- IBSS (independent basic service set) - когда 2 и более девайса соединены без участия AP (ad hoc network) Например apple AirDrop

- BSS (basic SS) - группа девайсов связанных через AP
В данной сети BSSID (basic servicw set ID) - испоьзуется для ункиальной идентификации AP. Это мак адрес точки доступа
BSA (basic service area) -область вокруг точки доступа, где есть сигнал

- ESS (extended service set) - несколько BSS обьединенных под одним SSID c помощью ethernet, но на разных каналах и с разными BSSID. Клиенты могут переключаться с одной AP на другую незаментно, это называется бесшовный роуминг
Чтобы коннект у клиентов не потерялся, между AP должен быть overlap порядка 10-15%

- MBSS (mesh basic service sets) - используется когда нет возможности соединить все точки доступа по ethernet. Они соединяются по wifi и имеют 2 сети. Одна BSS для клиентов и другая backhole network для сввязи между этиими AP
Тчока доступа связанная с интернетом нахзывается - RAP (root), а остальные - MAP (mesh)

- WGB (workgroup bridge) - клиент не поддерживающий wireless соединение подключается по ethernet к WGP AP и уже она подключается по WiFi к AP
- outdoor bridge

- passive scanning - хост слушает beacon messages от AP, которая периодически отсылает их в эфир
Далее идет аутентификация

        AP deployment methods

1) Autonomous - независимые, конфигурируются индивдуально, не зависят от WGP
AP в данной системе должны быть соединены со свичем транк-портом, даже если точка раздает один wi-fi. А все потому, что менеджмент траффик, используемвый для подключения к точке доступа должен ходить в отдельном влане и соответственно отдельной подсети.

2) Lightweight - часть функций (шифрование, отсылка beacons and probes) выполняется самой AP, а другая часть (managemrnt, authentication, security, association, roaming) - выполняется WLC
Это называется split-MAC architecture
WLC может находится в том же влане, или в другой сети
Через WLC (wireless LAN controller) конфигурируются все AP централизованно
LAPs use the Lightweight Access Point Protocol (LWAPP) to communicate with a WLAN controller (WLC
WLC handles RF management, Security/ QOS management, client authentication, client association/ roaming management
WLC и AP аутентифицируют дркг дркга по скртификатам X.509, установленным на каждом девайсе
Используется протокол CAPWAP:
 между WLC и AP формируется 2 тоннеля
 control tunnel (UDP 5246) - трафик зашифрован по дефолту
 data tunnel (UDP 5247) - по этому туннелю данные от клиентов AP идут до WLC (он инкапсулируется в новый заголовок, по типу как GRE, и получается туннель). Далее если тафик предназанчен хосту соединенному с той же AP, он возвращается обратно по этому же туннелю до AP, а если трафик предназнаен хосту в другой подсети, то он уходит с WLC через транк до хоста назначения.  Этот трафик не шифруется по дефолту, но можно зашифровать используя DTLS (это типа TLS для TCP, но только для UDP)
Тут транк порт не надо, в месте соединения AP и свича транк не надо, так как данные идут по туннелю в WLC, но нужен транк между WLC и свичем
Точка доступа в lightweight системе может функционровать в некольких режимах:
- local (In local mode, an AP creates two CAPWAP tunnels to the WLC.  One is for management, the other is data traffic.)
- flexConnect (если WLC сломался, то AP моежт гнать трафик напрямую через свич)
- sniffer (только ловит кажры и отправляет их на девайс работающий как wireshark)
- monitor (ловит кадры, чтобы найти вредоносный девайс)
- rogue detector (слушает LAN, а не радио эфир)
- SE-connect (сканирует спектр для поиска интерференции)
- bridge/mesh (устанавливает мост между удаленными AP)
- flex plus bridge (если WLC сломался, то AP моежт гнать трафик напрямую через свич)

3) Cloud-based
Это когда AP управляются из облака
Например Cisco Meraki
Но data трафик не идет в облако, как он шел при использовании WLC, а идет через свич напрямую куда надо

                WiFi Authentication methods

- open authentication (без пароля)
- WEP - это shared-key протокол (используеься один и тот же ключ) - не безопасен. Является так же и encryption методом.

- EAP (extensible auth protocol):
        Lightweight EP (LEP) - цисковский, это улучшенный WEP, уже тоже не безопасный
        EAP-FAST - аутентификация через TLS тунель, туннель поднимается после того как сервер отошлет клиенту PAC 
        PEAP - аутентификация через TLS тунель, туннель поднимается после проверки клиентом сертификата сервера
        EAP-TLS - для поднятия TLS туннеля требует наличие мертификата не только у клиента, но и у сервера. Но это требует много усилий, так как надо чтобы у каждого клиента быд сертификат. Поэтому большинство компаний использует PEAP. 

                WiFi Encryption and Integrity methods

TKIP (temporal key integrity protocol) - основан на WEP. Был создан как временное решение после того, как WEP стал уязвим. По сути это улучшенный WEP. Используется в WPA.
CCMP - используется в WPA2 и использует AES (advance evcryption standart) протокол шифрования, использует CBC-MAC для integrity (это разновидность MIC)
GCMP - используется в WPA3, так же использует AES и GMAC

                WPA (WiFi protected access)

Это стандарт созданный WiFI Alliance. Их 3 шт
Все они используют 2 способа аутентификации:
- personal - когда вводишь пароль (pre-shared key) для подключения
Он не передается в открытом виде. После аутентификации PSK используются для генерирования ключей шифрования, которыми будут шифроваться сообщения, чтобы другие хосты сети не могли увидеть содержимое сообщений.
- enterprise - исплользуется 802.1x RADIUS сервер

WPA3 использует AES для шифрования и SAE (Simultaneous Authentication of Equals) для аутентификации
WPA3 заменяет метод аутентификации с использованием предварительного общего ключа (PSK), используемый в WPA2, на SAE, который обеспечивает более надежную защиту от автономных атак по словарю и атак методом перебора. SAE позволяет пользователям безопасно проходить аутентификацию в сети, не раскрывая свои пароли, и защищает от различных уязвимостей, связанных с аутентификацией.

                Network automation

3 planes:
- data plane (пересылка данных)
- control plane (контроль пересылки данных, например построение таблиц маршрутизации)
- management plane (ssh, ntp)

Management Plane - For configuration device. CLI or GUI. Some protocol uses here: SSH, Telnel, SNMP, TFTP, SFTP, HTTPS. For example, you config router with Management place on CLI.

Control Plane - This is brane of device. This plane MAKE A DECİSİON. STP, ARP, OSPF, EIGRP, BGP used this plane for ROUTING.

Data Plane - Called also Forwarding Plane.It`s uses for Packet FORWARDING.

Generalization:
You config device Management Plane.
Device learn ROUTING path with Control Plane, and write it Routing Information Base (RIB) and Forwarding Information Base (FIB).
Data Plane use FIB for FORWARDING.

За процесс пересылки кадров на свиче отвечает ASIC а не CPU.
ASIC передает адрес назначение CAM таблице (это другое название mac-address table) и она возвращает асику интерфейс

Трафик control and management обрабатывается CPU

                                REST APIs

В основном используются Northbound интерфейсом в SDN архитектуре
CRUD (create, read, update, delete)

REST APIs обычно используют HTTP методы, но не всегда и это не обязательно
Create - POST
Read - GET
Update - PUT, PATCH
Delete - DELETE

Клгда HTTP клиент отсылает запрос HTTP серверу, то HTTP заголовок содержит следующую информацию:
- HTTP verb (method) - GET
- URI (Uniform resourse identifier)
А так же другие HTTP заголовки
Сервер отвесает клиенту отсылая RESPONSE на его REQUEST
- код статуса
1xx - informational ифнормационный ответ, типо запрос получен и оьбрабатывается (102 - processing)
2хх - successul - запрос получен и понятен и принят (200 - OK, 201 - created в ответ на POST request)
3хх - redirection (301 - moved permanently)
4хх - client error - или ошибка в написании адреса, или ресурса не существует (403 - unauthorized нужна аутентификация от клиента, 404 - not found )
5хх - server error - запрос получен и валиден, но сервер не может обработать по каким либо причсинам (500 - internal server error)

        REST (representational state transfer)

Так же изветен как REST-based APIs and RESTful APIs
REST описывает правила работы APIs. Это фреймворк для APIs

REST API использует клиент-скрверную архитектуру
Клиент использует API calls (как HTTP requests) для доступа к ресурсам сервера
Statelessness - згначит что каждый API вызов не зависим от других. Сервер не хранит инфу о предыдущих запросах, каждый запрос это новый  - event
REST APIs должны поддерживать кеширование данных
Но не все ресурсы должны кешироваться

                                        SDN (software-defined networking)

SDN это подход в котором уровень контроля находится в приложении, которое называется контроллер
Традиционно на каждом устройстве имеется свой control plane (свой OSPF, ACLs и тд)
В отличие от этого традиционного метода, SDN controller централизует все функции уровня контроля в одном месте (н-р калькуляция маршрутов)
В зависимости от типа решения применяемого для SDN, контроллер может выполнять все или часть функция уровня контроля
Controller взаимодействует с сетевыми девайсами используя APIs (через SBI)
Мы же взаимодействуем с контроллером испольуя какое либо приложение (application) общаясь с REST API контроллера (через NBI)
Application layer ----> Control layer ----> Infrastructure layer

overlay - виртуальная сеть поверх физической (н-р VXLAN)
fabric - совокупность 2х вышеперечисленных

- SD-WAN - SDN решение для автоматизации WAN

                        Configuration Management Tools

        Ansible
От Red Hat
Написан на питоне
Он agentless и использует SSH
Использует Push model
Использует playbooks написанные на YAML, inventory файл (написан используя INI или YAML), templates (в формате jinja2), variables (на YAML) которые подставляются в шаблон для создания завершенных файлов конфигурации

        Puppet
Написан на Ruby
Он agent-based
Не все цисковские девайсы поддерживают паппет-агента
Если клиент не поддерживает установку puppet агента, то в качестве прослойки можно использовать внешний агент, с которвм будет соединятся Puppet master и уже он будет через SSH соединятся с девайсом
Использует Pull model
Клиенты используют TCP 8140 для соединения с Puppet master
Использует не YAML, а свой проприетарный язык
Нужные файлы: manifest, templates.

        Chef
Написан на Ruby
Он agent-based
Не все цисковские девайсы поддерживают шеф-агента
Использует Pull model
TCP 10002
Использует язык DSL основанный на Ruby
Нужные файлы: resources, recipies, cookbooks, run-list
